We need a full mobile-first redesign and UX improvement of the website. This is not a small fix — this is a structured redesign to make the site look professional, modern, logical, and smooth on mobile devices.

The goal:
- clean, premium, service-based design
- similar UX level to Uber / Bolt / Deliveroo
- no messy layouts
- no repeated elements
- no broken mobile behaviour
- strong visual hierarchy
- easy and comfortable booking on phone
- polished, professional appearance

Please follow ALL instructions carefully and implement consistently.

-----------------------------------
4. ADDRESS AUTOCOMPLETE (CRITICAL)
-----------------------------------

This is one of the most important parts of the product and must feel smooth, fast, and reliable on mobile.

UI requirements:
- dropdown must appear directly under the input
- clean white background
- rounded corners
- no dark / black fullscreen overlay
- no page jumping or layout shifting
- clear spacing between items
- large tap targets (mobile-friendly)
- loading indicator inside input while searching
- clean “no results” state
- dropdown closes after selection

UX requirements:
- address selection must feel instant and smooth (like Uber / delivery apps)
- user must clearly understand they are selecting a real address, not typing freely
- Continue button must remain disabled until a valid selection is made

Data quality requirements:

The system should always TRY to return full addresses including:
- house / building number
- street name
- postcode
- city

Example of ideal result:
233 Alexandra Park Rd, London N22 7BJ, UK

Do NOT prioritize:
- postcode-only results
- area / borough names
- vague locations
- incomplete street-only results

Filtering:
- prioritize full UK addresses
- bias results towards London where relevant
- avoid irrelevant or overly broad results

Important balance (UX vs accuracy):

If the provider does NOT return a house number in some cases:
- DO NOT block the user completely
- allow selection of the closest valid address
- THEN require a follow-up field:

"House / Flat number" (required)

This ensures:
- the user is never stuck
- address remains complete and usable
- booking flow stays smooth

Final rule:
- user must NOT be able to continue with random typed text
- a real suggestion must be selected OR completed via house/flat field

Goal:
A fast, clean, mobile-friendly address experience that feels like a modern booking app, not a form.
